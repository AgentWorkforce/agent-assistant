### Workflow structure — `publish.yml`

Follows the relayfile/relayauth three-job pattern:

**Job 1: `build`** — Checkout → setup-node@v4 (node 22) → `npm ci` → run tests (all 4 packages) → typecheck (`tsc --noEmit` per package) → build (`npm run build` per package in DAG order) → version bump + sync → upload artifacts.

**Gates (resolves review finding #3):** The contract specifies four gates. The workflow implements them as four distinct steps, not collapsed:

| Gate | Workflow Step | Command |
|------|--------------|---------|
| Test | `Run tests — <pkg>` (×4) | `npx vitest run` per workspace |
| Typecheck | `Typecheck — <pkg>` (×4) | `npx tsc --noEmit` per workspace |
| Build | `Build packages` | `npm run build` per workspace in DAG order |
| Version sync | `Version all packages` | Inline node script syncing versions |

If any gate fails, the build job fails and no publish jobs run.

**Job 2: `publish`** — Matrix over four packages, `max-parallel: 1`, ordered: traits → core → sessions → surfaces. Download artifacts → `npm install -g npm@latest` → `npm publish --access public --provenance --tag <tag> --ignore-scripts`. Dry-run variant uses `--dry-run` without `--provenance`.

**Job 3: `create-release`** — Commit version bump → create git tag `v<version>` → `softprops/action-gh-release@v2`.

### Workflow structure — `ci.yml`

Triggers on `push` to main and `pull_request`. Runs: `npm ci` → test all 4 publishable packages → typecheck all 4 → build all 4. No publish, no version bump.

## 4. Workforce Profile Consumption — Direct Path via `@agentworkforce/workload-router`

### Corrected understanding (resolves review findings #4 and #5)

The review verdict identified that `BLOCKER-WF-001` is **incorrect**. The Workforce npm-provenance-publisher profile IS already consumable programmatically through the published `@agentworkforce/workload-router` package (v0.1.1):

```typescript
import { resolvePersona, materializeSkillsFor } from '@agentworkforce/workload-router';

const selection = resolvePersona('npm-provenance');
// selection.runtime — contains harness, model, systemPrompt, harnessSettings
// selection.skills — contains prpm/npm-trusted-publishing skill reference
// selection.tier — resolved tier based on routing profile

const skillPlan = materializeSkillsFor(selection);
// skillPlan — materialized skill instructions for the selected harness
```

The `personaCatalog` in `@agentworkforce/workload-router` includes the `npm-provenance` intent, which maps to the `npm-provenance-publisher.json` persona file. The persona is imported directly into the package source at build time.

### What this means for implementation

1. **Add `@agentworkforce/workload-router` as a root devDependency.** This provides typed, versioned access to the npm-provenance-publisher persona without copying any persona data locally.

2. **The publish workflow itself does not import the persona at runtime.** GitHub Actions YAML workflows cannot `import` npm packages. The persona's concrete technical requirements (permissions, flags, npm upgrade) are implemented directly in the workflow YAML with a source attribution comment referencing `@agentworkforce/workload-router` and the `npm-provenance` intent.

3. **The workflow orchestration layer** (TypeScript specify-publish-infrastructure workflow and any future CI validation scripts) CAN and SHOULD consume the persona programmatically:
   ```typescript
   import { resolvePersona } from '@agentworkforce/workload-router';
   const { runtime, skills } = resolvePersona('npm-provenance');
   ```

4. **OIDC trusted publishing (resolves review finding #5):** The Workforce persona specifies OIDC trusted publishing with "no long-lived NPM_TOKEN." The workflow MUST use GitHub Actions OIDC for npm provenance (`id-token: write`). The `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}` pattern is retained as a fallback only because npm trusted publisher OIDC setup requires per-package registration on npmjs.com (Task P1-4 in the implementation plan), which is an external manual step. The workflow comment MUST note that the target state is full OIDC with no `NPM_TOKEN`, pending trusted publisher registration.

### Harness implication

The Workforce persona defines three tiers (`best`, `best-value`, `minimum`), each specifying a harness and model for the publish agent. This is relevant for **workflow orchestration** (which harness runs the publish workflow), not for the GitHub Actions workflow itself. The `resolvePersona('npm-provenance')` call returns the appropriate tier based on the active routing profile, and the orchestration layer uses that to select the harness. The publish workflow YAML is harness-agnostic — it runs the same regardless of which harness invoked it.

### What is NOT done

- No `@agentworkforce/personas` package is created (unnecessary — workload-router already exports the persona)
- No raw GitHub URL fetch of persona JSON
- No copy of persona JSON or systemPrompt into this repo
- `BLOCKER-WF-001` is **retired** — it was based on incorrect analysis
